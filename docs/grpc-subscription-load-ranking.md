# Ранжирование gRPC-подписок по нагрузке на ноду

Цель: понять реальную стоимость каждого типа подписки для формирования тарифов.

---

## Быстрая сводка (от самого дешёвого к самому дорогому)

| # | Подписка | Нагрузка | Трафик/сек | CPU/msg | Множитель |
|---|----------|----------|------------|---------|-----------|
| 1 | Ping/Pong | Нулевая | ~20 B / 15 сек | 0 | **×1** (базовый) |
| 2 | Slots | Минимальная | ~400 B/s | O(1) | **×1** |
| 3 | Entries | Минимальная | ~5 KB/s | O(1) | **×1** |
| 4 | BlocksMeta | Низкая | ~0.1 KB/s | O(1) | **×2** |
| 5 | TransactionStatus | Низкая | ~50 KB/s | O(m) | **×3** |
| 6 | Transactions (фильтр) | Средняя | ~10-100 KB/s | O(m) | **×5** |
| 7 | Transactions (все) | Высокая | ~5-20 MB/s | O(1) | **×15** |
| 8 | Accounts (фильтр + slice) | Средняя | ~10-500 KB/s | O(n) | **×8** |
| 9 | Accounts (фильтр, без slice) | Высокая | ~1-50 MB/s | O(n) | **×20** |
| 10 | Accounts (все) | Очень высокая | ~100-500 MB/s | O(1) | **×50** |
| 11 | Blocks (с фильтром) | Очень высокая | ~10-50 MB/s | O(a×t) | **×80** |
| 12 | Blocks (все) | Экстремальная | ~50-200 MB/s | O(a+t) | **×200** |

> **m** = кол-во аккаунтов в фильтре, **n** = кол-во фильтров аккаунтов, **a** = аккаунтов в блоке, **t** = транзакций в блоке

## Подробный разбор каждого типа

---

### 1. SLOTS — Обновления слотов

**Что получает клиент:** номер слота, родительский слот, статус (Processed / Confirmed / Finalized), ошибка (для Dead).

**Размер сообщения:** ~50-100 байт

**Частота:** ~2-4 раза в секунду на каждый слот (processed → confirmed → finalized), плюс промежуточные статусы (FirstShredReceived, CreatedBank, Completed). Итого ~10-20 сообщений/сек.

**Фильтры:**
| Фильтр | Что делает | Стоимость фильтрации |
|--------|-----------|---------------------|
| `filter_by_commitment` | Только определённый commitment level | O(1), одна проверка |
| `interslot_updates` | Включить/выключить промежуточные статусы | O(1), одна проверка |

**Нагрузка на ноду:**
- CPU: практически нулевая
- Память: ~1 KB в очереди
- Сеть: ~400 B/s
- **Самая дешёвая подписка**

**Тариф: базовый / бесплатный** — стоимость обслуживания околонулевая.

---

### 2. ENTRIES — Записи внутри блока

**Что получает клиент:** слот, индекс entry, кол-во хэшей, хэш (32 байта), кол-во исполненных транзакций, стартовый индекс транзакции.

**Размер сообщения:** ~80-120 байт

**Частота:** ~50-100 entry/сек (зависит от нагрузки на сеть Solana). Каждый слот содержит десятки entries.

**Фильтры:** Нет дополнительных фильтров — только по имени подписки. Если подписан — получаешь все.

**Нагрузка на ноду:**
- CPU: минимальная
- Память: ~10 KB в очереди
- Сеть: ~5-10 KB/s
- **Очень дешёвая подписка**

**Тариф: базовый** — мало кто использует, данные мелкие.

---

### 3. BLOCKS_META — Метаданные блоков

**Что получает клиент:** слот, blockhash, parent blockhash, parent slot, block_time, block_height, rewards (включая все вознаграждения валидаторов), executed_transaction_count, entries_count.

**Размер сообщения:** ~200-500 байт (основной вес — rewards, может быть список из сотен записей).

**Частота:** ~2-3 в секунду (один раз per slot × commitment level).

**Фильтры:** Нет — только по имени подписки.

**Нагрузка на ноду:**
- CPU: минимальная
- Память: ~2 KB в очереди
- Сеть: ~0.5-1.5 KB/s
- **Дешёвая подписка**

**Тариф: базовый** — маленькие сообщения, редкие.

---

### 4. TRANSACTION_STATUS — Статус транзакций (облегчённый)

**Что получает клиент:** подпись транзакции (64 байта), ошибка (если есть), слот. **Без тела транзакции** — только результат.

**Размер сообщения:** ~100-200 байт

**Частота на mainnet:** ~3000-5000 транзакций/сек (все). С фильтрами — намного меньше.

**Фильтры (идентичны Transactions):**
| Фильтр | Что делает | Стоимость |
|--------|-----------|----------|
| `vote` | Включить/исключить голосования | O(1), проверка bool |
| `failed` | Включить/исключить неудачные | O(1), проверка bool |
| `signature` | Конкретная подпись | O(1), сравнение 64 байт |
| `account_include` | Хотя бы один из аккаунтов | O(m), пересечение HashSet |
| `account_exclude` | Ни одного из этих аккаунтов | O(m), пересечение HashSet |
| `account_required` | Все указанные аккаунты | O(m), проверка подмножества |

**Нагрузка на ноду:**
- CPU: низкая (фильтрация по HashSet быстрая)
- Память: ~50 KB в очереди (при фильтре)
- Сеть: ~10-50 KB/s (с фильтром по аккаунту — единицы KB/s)
- **Дешёвая подписка**, но нужна фильтрация каждой транзакции

**Тариф: базовый+** — данные мелкие, но фильтрация проходит по каждой транзакции в сети.

---

### 5. TRANSACTIONS (с фильтром) — Полные транзакции по фильтру

**Что получает клиент:** полное тело транзакции — подпись, все аккаунты, инструкции, мета (логи, балансы до/после, inner instructions), slot, index.

**Размер сообщения:** ~1-10 KB (типичная), до ~1 MB (максимум).

**Частота на mainnet:**
- Все: ~3000-5000/сек
- Без голосований: ~1500-2500/сек
- По одному аккаунту: ~1-100/сек (зависит от популярности)

**Фильтры:** те же что и TransactionStatus (см. выше).

**Сценарии по нагрузке:**

| Фильтр | Что делает нода | Трафик клиенту | Множитель |
|--------|----------------|---------------|-----------|
| `account_include: [1 ключ]` | Проверяет HashSet для каждой tx | ~1-100 KB/s | ×5 |
| `account_include: [100 ключей]` | То же, но пересечение чуть дороже | ~10 KB-1 MB/s | ×8 |
| `vote: false` | Фильтрует ~50% транзакций | ~5-15 MB/s | ×12 |
| Без фильтра (все) | Никакой фильтрации, копирует все | ~10-30 MB/s | ×15 |

**Ключевой момент:** фильтрация происходит **для каждой транзакции в сети**, даже если клиент получит 1 из 5000. Нода тратит CPU на проверку всех 5000.

**Тариф: средний** — CPU-нагрузка пропорциональна общему потоку транзакций, а не количеству совпадений.

---

### 6. ACCOUNTS (с фильтром + data slice) — Аккаунты с вырезкой данных

**Что получает клиент:** pubkey, owner, lamports, executable, rent_epoch + **только указанные байтовые диапазоны** из данных аккаунта (вместо полных данных).

**Пример:** аккаунт Token Account занимает 165 байт. Если нужен только баланс (8 байт по offset 64), клиент указывает `data_slice: [{offset: 64, length: 8}]` и получает 8 байт вместо 165.

**Размер сообщения:** ~100 байт (метаданные) + размер slice (обычно 8-200 байт). Итого ~100-400 байт.

**Частота на mainnet:** ~50 000-100 000 обновлений аккаунтов/сек (все). С фильтром по pubkey — единицы/сек.

**Фильтры:**
| Фильтр | Что делает | Стоимость |
|--------|-----------|----------|
| `account: [pubkeys]` | Конкретные адреса аккаунтов | O(1) HashSet lookup |
| `owner: [pubkeys]` | По владельцу (напр. Token Program) | O(1) HashSet lookup |
| `memcmp: {offset, data}` | Совпадение байтов в данных | O(data.len), до 128 байт |
| `datasize: N` | Точный размер данных | O(1) |
| `token_account_state` | Валидный SPL Token аккаунт | O(data.len) |
| `lamports: {eq/ne/lt/gt}` | Фильтр по балансу | O(1) |
| `nonempty_txn_signature` | Есть ли привязанная транзакция | O(1) |

**Сценарии:**

| Фильтр | Нагрузка | Трафик | Множитель |
|--------|---------|--------|-----------|
| `account: [1 pubkey]` + slice | Минимальная | ~100 B/s | ×3 |
| `account: [100 pubkeys]` + slice | Низкая | ~10 KB/s | ×5 |
| `owner: [Token Program]` + slice | Средняя | ~100 KB-1 MB/s | ×8 |
| `owner: [Token Program]` + memcmp + slice | Средняя | ~10-100 KB/s | ×8 |

**Тариф: средний** — data slice критически снижает трафик. Без slice стоимость гораздо выше (см. ниже).

---

### 7. ACCOUNTS (с фильтром, БЕЗ data slice) — Аккаунты полностью

**То же что выше, но клиент получает ВСЕ данные аккаунта.**

**Размер сообщения:**
- Пустой аккаунт: ~100 байт
- Token Account: ~265 байт (165 данных + метаданные)
- Mint Account: ~182 байта
- AMM Pool: ~1-10 KB
- Программный аккаунт: ~1-2 MB (data = весь код программы)

**Проблемные сценарии:**

| Фильтр | Трафик | Проблема | Множитель |
|--------|--------|---------|-----------|
| `owner: [Token Program]` без slice | ~10-50 MB/s | Тысячи аккаунтов по 165B | ×20 |
| `owner: [Raydium AMM]` без slice | ~5-30 MB/s | Сотни аккаунтов по KB | ×20 |
| `account: [1 pubkey]` без slice | ~1 KB/s | Нормально, но зачем без slice? | ×5 |

**Тариф: высокий** — трафик может быть огромным. Рекомендовать клиентам использовать data slice.

---

### 8. ACCOUNTS (без фильтра — ВСЕ аккаунты)

**Клиент получает КАЖДОЕ обновление КАЖДОГО аккаунта.**

**Размер потока:** ~100-500 MB/s непрерывно.

**Нагрузка:**
- CPU: низкая (нет фильтрации), но высокая на сериализацию
- Память: 16 MB очередь **постоянно заполнена**, сообщения дропаются
- Сеть: клиент физически не успевает принимать
- **Практически бесполезная подписка** — клиент почти гарантированно отстанет

**Тариф: запретительно высокий / не предлагать** — дестабилизирует ноду, забивает канал.

---

### 9. BLOCKS (с фильтром по аккаунту)

**Что получает клиент:** полный блок — ВСЕ транзакции + ВСЕ аккаунты + ВСЕ entries + метаданные блока. С фильтром `account_include` — только транзакции/аккаунты, затрагивающие указанные адреса.

**Размер сообщения:** ~1-50 MB на блок (зависит от фильтра).

**Частота:** ~2-3 блока/сек (по одному на commitment level).

**Стоимость сборки блока:**
```
Для каждого блока:
  Для каждого аккаунта в блоке (~1000-10000):
    Проверить: входит ли pubkey/owner в account_include?
  Для каждой транзакции в блоке (~2000-5000):
    Для каждого аккаунта транзакции (~5-30):
      Проверить пересечение с account_include
```
Итого: **O(accounts + transactions × accounts_per_tx)** = O(10 000 + 5 000 × 20) = ~110 000 операций на блок.

**Ограничения:**
- **Replay запрещён для блоков** (слишком дорого собирать исторические блоки)
- `include_transactions`, `include_accounts`, `include_entries` — можно выключить части

**Сценарии:**

| Вариант | Трафик | CPU | Множитель |
|---------|--------|-----|-----------|
| Блок с 1 account_include, только tx | ~100 KB-1 MB/block | Высокий | ×40 |
| Блок с 100 account_include, всё | ~5-20 MB/block | Высокий | ×60 |
| Блок без include_accounts (только tx+meta) | ~1-10 MB/block | Средний | ×30 |

**Тариф: очень высокий** — каждый блок требует полного обхода всех транзакций и аккаунтов.

---

### 10. BLOCKS (без фильтра — ВСЕ данные)

**Клиент получает ПОЛНЫЙ блок: все аккаунты, все транзакции, все entries.**

**Состав одного блока на Solana mainnet:**

| Компонент | Количество | Размер каждого | Итого |
|---|---|---|---|
| Транзакции | ~2000-5000 | ~1-5 KB | ~10-25 MB |
| Аккаунты (обновлённые) | ~5000-15000 | ~100 B - 2 MB | ~20-100+ MB |
| Entries | ~50-100 | ~100 B | ~5-10 KB |
| BlockMeta | 1 | ~400 B | ~400 B |
| **ИТОГО за блок** | | | **~30-130 MB** |

**Частота:** новый блок каждые ~400 мс (2.5 блоков/сек).

**Непрерывный поток:** 130 MB × 2.5 = **~325 MB/s**

**Почему это убивает ноду:**

```
Очередь клиента:  16 MB (messages_len_max по умолчанию)
Один блок:        ~100 MB

→ Блок НЕ ВЛЕЗАЕТ в очередь → дропается
→ Клиент получает дырки в данных
→ Нода тратит CPU на сборку и сериализацию впустую

Даже с увеличенной очередью (256 MB):
  Клиент на 1 Gbps (125 MB/s) не успевает скачать 100 MB за 400 мс
  → нужно 800 мс → каждый второй блок дропается
  → нужен минимум 10 Gbps канал чтобы ОДИН клиент не отставал
```

**Сравнение с другими подписками:**
```
Slots:              ~400 B/s     ← 1 000 000× дешевле
Accounts (1 key):   ~100 B/s    ← 3 000 000× дешевле
Transactions:       ~15 MB/s    ← 20× дешевле
Blocks (все):       ~325 MB/s   ← ЭТО
```

**Нагрузка:**
- CPU: высокая (сборка + сериализация гигантского protobuf ~100 MB)
- Память: 16 MB очередь мгновенно переполнена, данные дропаются
- Сеть: ~325 MB/s = 2.6 Gbps на ОДНОГО клиента
- **Самая дорогая подписка в системе, по сути "скачать всю Solana в реальном времени"**

**Тариф: запретительно высокий / не предлагать** — один такой клиент может деградировать ноду. Если нужны полные блоки — клиенту нужна своя нода или локальный доступ через SHM.

## Что дороже всего для ноды (разложение по ресурсам)

### CPU (фильтрация)

```
Каждое сообщение проходит через фильтр КАЖДОГО клиента отдельно.
100 клиентов = 100× проверка фильтра для каждого сообщения.

Стоимость одной проверки:
  Slot:          ~10 нс (2 сравнения)
  Entry:         ~10 нс (проверка имени)
  BlockMeta:     ~10 нс (проверка имени)
  TxStatus:      ~50-200 нс (HashSet пересечения)
  Transaction:   ~50-200 нс (HashSet пересечения)
  Account:       ~50-500 нс (HashSet + memcmp + data filters)
  Block:         ~100 µs-1 мс (обход всех tx и accounts в блоке)
```

### Память

```
Каждый клиент: 16 MB очередь (messages_len_max)
100 клиентов = 1.6 GB только на очереди

+ буферизация сообщений (messages_cached_max = 1024 per commitment × 3 levels)
+ Arc-указатели на общие данные (данные не дублируются, но метаданные — да)
```

### Сеть (bandwidth)

```
Основной ограничитель. Типичная нода имеет 1-10 Gbps.

1 Gbps = ~125 MB/s
10 клиентов с "все транзакции" = 10 × 15 MB/s = 150 MB/s → забивает 1 Gbps
1 клиент с "все блоки" = 100 MB/s → 80% канала
```

### Диск / SHM

```
Только для SHM-based клиентов (richat-server как reader).
Пре-фильтр V2 снижает нагрузку на чтение:
  Bloom skip: читает только 128 байт index entry вместо KB данных
  Для 99% транзакций при фильтре по 1 аккаунту → экономия 1000× на I/O
```

---

## Рекомендации по защите ноды

1. **Лимитировать `any: true`** (пустые фильтры) — это самые дорогие подписки
2. **Обязывать data_slice для account подписок по owner** — снижает трафик в 10-100×
3. **Ограничить max клиентов с Block подписками** — 1-2 максимум
4. **Не предлагать Blocks без account_include** — неподъёмная нагрузка
5. **Ограничить account_include_max** — защита от клиентов с 10 000 ключей в фильтре
6. **Rate limit на replay** — исторические данные дороже live (replay queue 256 MB)
7. **Мониторить messages_len_max** — если очередь клиента постоянно полная, клиент не успевает и тратит ресурсы впустую

## Формула оценки стоимости подписки

```
Стоимость ≈ (частота_сообщений × процент_match × размер_сообщения × кол-во_клиентов)
           + (частота_сообщений × стоимость_фильтрации × кол-во_клиентов)

Где:
  частота_сообщений = сколько сообщений этого типа в секунду в сети
  процент_match     = какая доля пройдёт фильтр (1% для узкого фильтра, 100% для пустого)
  размер_сообщения  = средний размер одного совпавшего сообщения
  стоимость_фильтра = CPU на проверку одного сообщения
```

| Тип | Частота/сек | Размер | Стоимость фильтра |
|-----|------------|--------|-------------------|
| Slot | ~15 | 80 B | 10 нс |
